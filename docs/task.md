# vurl 実装計画

## 1. 前提

- Git 操作はユーザーが行う
- Rust は macOS にインストール済みである
- Node.js は `nodenv` で 24 系 LTS を使用する
- 開発中の定義ルートは `--root` で切り替えられる前提とする

## 2. ゴール

最初の到達点は、`vurl-backend --root <dir>` で起動し、React フロントエンドから以下ができる状態とする。

- プロジェクト一覧取得
- 環境一覧取得
- リクエスト定義ツリー表示
- 定義読込
- リクエスト編集
- 送信
- レスポンス表示
- 認証
- ログ出力

## 3. 実装方針

- 先に Rust 側のデータ構造とファイル読込を固める
- 次に HTTP API を作る
- その後 React 側を載せる
- 認証、自動リトライ、ログはバックエンド責務に寄せる
- YAML スキーマに対応する Rust 構造体を単一の基準にする

## 4. 開発環境セットアップ

### 4.1 Node.js

- `nodenv` で 24 系 LTS を選ぶ
- `.node-version` を置いてバージョンを固定する
- パッケージマネージャは `npm` を前提にしてよい

### 4.2 Rust

- バックエンドは Cargo 管理とする
- プロダクションバイナリ名は `vurl-backend` とする
- HTTP サーバは Actix Web を使う

## 5. 実装フェーズ

## 5.1 フェーズ 1: リポジトリ初期化

作業:

- フロントエンドプロジェクトを作成する
- Rust バックエンドプロジェクトを作成する
- `docs/` と `sample/` を現構成のまま保持する
- `.node-version` を追加する

成果物:

- React アプリの雛形
- `vurl-backend` の Cargo プロジェクト
- 開発時の起動手順

## 5.2 フェーズ 2: Rust のスキーマ定義

作業:

- リクエスト YAML 用構造体を作る
- 環境 YAML 用構造体を作る
- 認証 YAML 用構造体を作る
- Serde で YAML をパースできるようにする
- バリデーション関数を作る

対象:

- `requests/*.yaml`
- `environments/*.yaml`
- `environments/auth.yaml`

成果物:

- Rust の型定義
- YAML ローダ
- バリデーション結果型

## 5.3 フェーズ 3: 定義ルート読込

作業:

- `--root` を受け取る CLI 引数を実装する
- `{root}/defs` と `{root}/logs` を解決する
- プロジェクト一覧を列挙する
- リクエストツリーを構築する
- 環境一覧を構築する
- `c` 用の全 YAML チェック処理を作る

成果物:

- ルートディレクトリ解決
- プロジェクト走査
- 全体チェック処理

## 5.4 フェーズ 4: Actix API

作業:

- `GET /api/projects`
- `GET /api/environments`
- `GET /api/tree`
- `GET /api/definition`
- `POST /api/send`
- `POST /api/logs/new`
- `GET /api/runtime`

補足:

- `POST /api/projects/select` は不要か再確認する
- アクティブプロジェクトはクライアントが毎回指定する設計でもよい

成果物:

- フロントエンドが必要とするバックエンド API

## 5.5 フェーズ 5: リクエスト実行基盤

作業:

- 変数展開を実装する
- 未定義変数エラーを実装する
- query/header/body の送信処理を実装する
- `json` と `form` の body 送信を実装する
- VPN 切替後も毎回新しい通信条件で送信できるようにする

成果物:

- 実リクエスト送信機能
- 変数展開機能
- 送信前バリデーション

## 5.6 フェーズ 6: 認証

作業:

- `auth: true/false` を解釈する
- `fixed` モードを実装する
- `default` マッピングを実装する
- `http` モードを実装する
- `response.inject` を環境変数更新として実装する
- `401/403` 時の再認証 + 1 回リトライを実装する

成果物:

- 認証実行器
- 再認証リトライ

## 5.7 フェーズ 7: ログ

作業:

- JST ベースの日次ログ切替を実装する
- 手動新規ログ作成を実装する
- `.md` 出力を実装する
- 1 リクエスト 1 コードフェンス形式を実装する
- マスク値置換を実装する

成果物:

- ログファイル作成器
- Markdown ログフォーマッタ

## 5.8 フェーズ 8: 親子プロセス構成

作業:

- 親プロセスで標準入力を監視する
- `c` で全 YAML チェックを実行する
- `r` でバックエンド子プロセスのみ再起動する

成果物:

- 親プロセスランチャ
- 子プロセス再起動機構

## 5.9 フェーズ 9: React UI

作業:

- 3 ペインレイアウトを作る
- サイドバーのフィルタとツリー表示を作る
- 環境切替 UI を作る
- リクエスト編集 UI を作る
- `form` の key-value 編集 UI を作る
- `json` のフリーテキスト編集 UI を作る
- レスポンス表示 UI を作る
- 認証 ON/OFF と ID/PW 入力 UI を作る
- 新規ログファイルボタンを作る

成果物:

- 実用最小 UI

## 5.10 フェーズ 10: zsh 連携

作業:

- `vurl.zsh` を作る
- `vurl` 関数を実装する
- 引数なしで `vurl-backend` 起動 + `localhost` を open する
- `-l` でログディレクトリへ `cd` する
- `-y` で定義ディレクトリへ `cd` する
- `--root` の固定値を zsh 側で差し込めるようにする

成果物:

- `vurl.zsh`
- ローカル運用用コマンド

## 7. 実装上の注意

- Git は触らない
- YAML 編集機能は UI に入れない
- ログには `mask` がある変数だけマスク値を出す
- `mask` 未指定の値はそのまま出る
- 未定義変数は送信前エラーにする
- `c` は全プロジェクト全 YAML をチェックする
- `r` はバックエンドのみ再起動する

## 8. 先に切るとよいタスク

1. `.node-version` を `nodenv` 向けに追加する
2. Cargo プロジェクトを `vurl-backend` 名で初期化する
3. React プロジェクトを作る
4. sample YAML を読む Rust テストを書く
5. `--root` で `sample/` を読める CLI を作る
