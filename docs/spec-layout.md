# Layout Spec

## 目的

`docs/memo.md` の以下をまとめて扱う。

- 1. 画面レイアウトを調整したい
- 2. 画面が大きすぎて収まっていない
- 3. `input password` を `text` にしたい
- 4. Enter で送信できるようにしたい
- 5. ブラウザで複数タブを使ってリクエストを投げたい
- 18. レスポンスが画像のときは画像を表示したい
- 19. レスポンスが JSON のときは整形したい

本書は実装前の基本方針と、修正後レイアウト案を定義する。

## 基本方針

- 画面全体は `100vw x 100vh` に収める。
- `html`, `body`, `#root`, アプリの最上位コンテナは `height: 100%` を前提にする。
- ページ全体のスクロールは発生させない。
- スクロールは各ペイン内部だけで発生させる。
- 左サイドバー、中リクエスト、右レスポンスの 3 ペイン構成は維持する。
- 各ペインは `min-height: 0` を明示し、内部スクロールを成立させる。
- ボタンやレスポンス本文が画面外にはみ出さないよう、固定ヘッダ + 可変本文の構造にする。
- モバイル幅では縦積みにしてもよいが、その場合も「画面全体スクロール禁止、各セクション内スクロール」を維持する。

## 修正後レイアウト案

### 全体

- 左: サイドバー
- 中: リクエスト
- 右: レスポンス

デスクトップでは 3 カラム固定にする。

- サイドバー: `280px` 前後の固定幅
- リクエスト: 可変幅
- レスポンス: 可変幅

中央と右はそれぞれ以下の 2 段構成にする。

- 上段: スクロールしない固定ヘッダ領域
- 下段: `overflow: auto` の本文領域

これにより以下を解消する。

- サイドバーが長すぎて縦に伸びる
- Send ボタンが見切れる
- Response Body が見切れる

### サイドバー

上から順に以下を置く。

- `Project`
- `Environment`
- `Filter`
- `New Log File`
- 現在のログファイル表示
- リクエストツリー

レイアウト方針:

- 上部のフォーム群は固定高さのヘッダブロックにまとめる
- リクエストツリーだけを残り領域いっぱいに広げる
- ツリー領域のみ縦スクロールにする

これで、定義数が増えてもページ全体は伸びない。

### リクエストペイン

上部固定領域:

- `NAME` は read only のテキスト表示にする
- `METHOD` は YAML 定義値をそのまま read only 表示にする
- `PATH`
- 認証設定
- `Send` ボタン

下部スクロール領域:

- `Query`
- `Headers`
- `Body`

`Query / Headers / Body` は同時表示ではなくタブ表示にする。

- タブは 1 つだけ表示
- タブ内容だけがスクロールする
- `Body` は YAML に定義された形式をそのまま表示・編集する
- `form/json` の切り替え UI は廃止する

補足:

- `NAME`, `METHOD` は送信対象の定義情報として見せるだけにする
- 編集可能な主要入力は `PATH` と認証まわりとアクティブタブの内容に絞る
- `Send` ボタンは固定領域に置き、常に見える状態にする

`Body` の扱い:

- YAML が `json` なら JSON テキストエリアを表示する
- YAML が `form` なら form の key-value editor を表示する
- ユーザが画面上で `json / form` を切り替えることはできない

### 認証 UI

以下に変更する。

- `INPUT` の初期値は `preset`
- `preset` が存在しない environment のときだけ `manual` にフォールバックする
- `preset` のときは `Preset` セレクトのみ表示する
- `preset` のときは `ID`, `PASSWORD` を非表示にする
- `manual` のときだけ `ID`, `PASSWORD` を表示する
- `PASSWORD` は `password` ではなく `text` にする

### Enter 送信

- リクエストペイン全体を `form` として扱う
- 単一行 input 上で Enter を押したら送信する
- `textarea` 内の Enter は改行として扱い、送信しない

### 複数タブ対応

複数ブラウザタブで独立して操作できることを前提にする。

- タブごとにフロントの編集中 state を保持する
- 初期表示時に他タブの state を上書きしない
- 永続化が必要な場合も `localStorage` の単一キー共有は避ける

今回の UI 改修では、少なくとも以下を守る。

- フロント state をグローバル共有しない
- タブを増やしても既存タブの編集中内容に影響しない

サーバ側に「最後に送信した project / environment」を保存する対応は別件で扱う。

### レスポンスペイン

上部固定領域:

- `Status`
- `Retried Auth`

下部スクロール領域:

- `Headers`
- `Body`

レスポンスも `Headers / Body` のタブ表示にする。

`Body` タブの表示ルール:

- `Content-Type` が JSON 系なら整形して表示する
- `Content-Type` が画像系なら画像プレビューを表示する
- それ以外はプレーンテキスト表示にする

想定する判定例:

- JSON: `application/json`, `application/*+json`
- Image: `image/png`, `image/jpeg`, `image/webp`, `image/gif`, `image/svg+xml`

## 実装方針

### フロントエンドのみで完結する変更

- 全体レイアウトの再構成
- ペイン内スクロール化
- `NAME`, `METHOD` の read only 表示化
- `INPUT` の初期値を `preset` に変更
- `preset` 時の `ID`, `PASSWORD` 非表示化
- `PASSWORD` を `text` に変更
- `Query / Headers / Body` のタブ化
- レスポンス `Headers / Body` のタブ化
- Enter 送信
- JSON レスポンス整形

### API 追加またはレスポンス表現の見直しが必要な変更

- 画像レスポンスの正確な表示

レスポンス `Body` は Response Header に応じて以下を切り替える。

- JSON: 整形表示
- text: プレーンテキスト表示
- image: 画像プレビュー表示

現状の `SendResponse` は `body: string` しか返していないため、画像プレビューを安定して行うには追加情報が必要になる可能性が高い。

候補:

- 画像レスポンス時だけ base64 を別フィールドで返す
- `body_encoding` のようなメタ情報を返して data URL 化可能にする
- 画像レスポンス専用の取得経路を用意する

少なくとも以下のどちらかは必要。

- バイナリを壊さずフロントへ渡す仕組み
- 画像レスポンス専用の取得経路

## 実装順の提案

1. レイアウト再構成とペイン内スクロール化
2. リクエスト UI の簡素化
3. 入力タブとレスポンスタブの導入
4. Enter 送信対応
5. JSON レスポンス整形
6. 画像レスポンス表示のための API 見直し

## 確定前提

- `METHOD` は YAML 定義値を read only 表示する
- リクエスト `Body` の形式は YAML に従い、画面で切り替えない
- レスポンス `Body` は Response Header に応じて JSON, text, image を切り替えて表示する
- 画像表示に必要であれば API 形式の変更を行う
